🚨 Critical Vulnerabilities

1. Hardcoded Test Credentials in Production Code

CRITICAL

Location: src/contexts/AuthContext.tsx (lines 33-37)

Issue: Demo user credentials are hardcoded directly in source code:

const DEMO_USERS = [
  { email: 'student@demo.com', password: 'demo123', role: 'student' },
  { email: 'parent@demo.com', password: 'demo123', role: 'parent' },
  { email: 'admin@pathpilot.com', password: 'admin123', role: 'admin' },
];

Risk:

  • Credentials visible in git history permanently
  • Anyone with repository access has test account access
  • If these accounts were used in staging, real accounts compromised
  • Simple passwords make brute force attacks trivial
✓ Immediate Actions:
  1. Remove DEMO_USERS array immediately
  2. Move to environment variables or separate test fixture file
  3. Rotate any credentials used for test accounts
  4. Add .gitignore rule to prevent future leaks: *.test.ts, *.fixture.ts
  5. Run git filter-branch to scrub history (if repo is fresh)

2. localStorage User Data Persistence Without Encryption

CRITICAL

Location: src/contexts/AuthContext.tsx (multiple locations)

Issue: Complete user objects including sensitive data stored in plaintext localStorage:

localStorage.setItem('pathpilot_user', JSON.stringify(newUser));
// Stored data includes: id, email, name, role, approval_status, education_stage

Risk:

  • localStorage is accessible to any script on the domain (XSS attacks)
  • No encryption - attackers can read user data directly
  • Any browser extension with permissions can access
  • Violates GDPR/privacy regulations (unencrypted PII)
  • Mobile browsers may sync to cloud unencrypted
✓ Immediate Actions:
  1. Implement XSS protection: Content Security Policy (CSP) headers
  2. Never store sensitive user data in localStorage
  3. Use httpOnly, Secure cookies for session tokens only
  4. Implement encryption if localStorage must be used: tweetnacl-js, crypto-js
  5. Store only session token in localStorage, fetch full profile from API

3. Missing Environment Variable Validation

CRITICAL

Location: src/lib/supabase.ts

Issue: Supabase credentials retrieved without validation or error handling:

const supabaseUrl = import.meta.env.VITE_SUPABASE_URL;
const supabaseAnonKey = import.meta.env.VITE_SUPABASE_ANON_KEY;

export const isSupabaseConfigured = Boolean(supabaseUrl && supabaseAnonKey);
// If keys are missing, silently falls back to localStorage

Risk:

  • If env vars not set, app silently falls back to insecure localStorage
  • No warning logged about security implications
  • Developers may not realize they're using fallback in production
  • Anonkey exposed on client-side (acceptable for Supabase but needs RLS rules)
✓ Immediate Actions:
  1. Validate env vars at app startup
  2. Throw error if critical vars missing in production
  3. Log warnings if fallback mode active
  4. Never allow fallback in production builds
  5. Use environment-specific .env files: .env.production, .env.development

4. No CSRF Protection on State-Changing Operations

CRITICAL

Location: All API calls in src/contexts/

Issue: No CSRF tokens on mutations (signUp, updateProfile, etc.)

Risk:

  • Attacker can craft malicious website that performs actions on behalf of user
  • User visits attacker site while logged into PathPilot
  • User's profile could be modified without knowledge
  • Financial transactions (if implemented) could be exploited
✓ Immediate Actions:
  1. Implement CSRF token generation on server (Supabase Edge Functions)
  2. Validate CSRF tokens on all state-changing requests
  3. Use SameSite cookie attribute: SameSite=Strict
  4. Implement rate limiting on sensitive endpoints

5. Insufficient Input Validation & Sanitization

CRITICAL

Location: All form handlers and data processing

Issue: No visible input validation or sanitization in components

Risk:

  • SQL Injection (if queries built improperly on backend)
  • Cross-Site Scripting (XSS) if user input rendered directly
  • NoSQL Injection attacks
  • Buffer overflow in text fields
  • Malformed data corrupts database
✓ Immediate Actions:
  1. Implement Zod or Yup schema validation on all forms
  2. Sanitize HTML output: use DOMPurify or similar
  3. Never execute user input as code
  4. Use prepared statements (Supabase uses parameterized queries)
  5. Implement content security policy (CSP)

🔐 Authentication Security

Password Policy Issues

Weak Password Requirements

HIGH

Issue: No minimum password complexity requirements visible in signup

Risk: Users can create weak passwords (e.g., "password123")

✓ Remediation:
  • Enforce minimum 12 characters in production
  • Require mixed case (uppercase + lowercase)
  • Require numbers and special characters
  • Check against common password lists (HaveIBeenPwned API)
  • Display password strength meter to users

Session Management Issues

No Session Timeout Implementation

HIGH

Issue: Users remain logged in indefinitely

Risk: If device is compromised, attacker has permanent access

✓ Remediation:
  • Implement 30-minute idle timeout
  • Force re-authentication after 24 hours
  • Show warning before session expires
  • Allow users to configure timeout preferences

No Multi-Device Session Management

HIGH

Issue: No way to view or revoke sessions from other devices

Risk: Compromised old devices retain access indefinitely

✓ Remediation:
  • Implement session tracking table in database
  • Show active sessions in settings page
  • Allow users to logout all other sessions
  • Track device info: browser, OS, IP address

Two-Factor Authentication (2FA)

2FA Not Implemented

HIGH

Issue: No second factor authentication for sensitive operations

Risk: Password compromise leads to complete account takeover

✓ Remediation:
  • Implement TOTP (Time-based One-Time Password) with authenticator apps
  • Offer SMS/Email backup codes
  • Require 2FA for admin accounts (mandatory)
  • Make 2FA optional but recommended for regular users
  • Use Supabase Auth's built-in 2FA support

🛡️ Data Protection & Privacy

Data Encryption

Sensitive Data Not Encrypted in Transit

HIGH

Issue: No visible HTTPS enforcement or certificate pinning

Risk: Man-in-the-middle (MITM) attacks can intercept user data

✓ Remediation:
  • Enforce HTTPS only: add HSTS header
  • Use certificate pinning for API calls
  • Disable mixed-content: all resources must be HTTPS
  • Regular SSL/TLS certificate audits

Personally Identifiable Information (PII) Not Encrypted at Rest

HIGH

Issue: Student data (name, city, state, phone) stored unencrypted in database

Risk: Database breach exposes all student personal information

✓ Remediation:
  • Encrypt sensitive fields at database level
  • Use Supabase's field-level encryption (pgcrypto extension)
  • Implement key management service (AWS KMS, Azure Key Vault)
  • Separate encryption keys from application code

COPPA Compliance (Children's Online Privacy)

MEDIUM

Issue: Platform serves students age 8-18, requires special protections

Risk: FTC violations if student data collected without parental consent

✓ Remediation:

  • Implement parent verification system
  • Restrict data collection from minors
  • Clear privacy policy in child-friendly language

🏢 Infrastructure Security

Supabase Row-Level Security (RLS) Configuration Unknown

MEDIUM

Issue: Cannot verify RLS policies from frontend code alone

Risk: If RLS not properly configured, users can access other's data

✓ Required Actions:
  • Audit all Supabase RLS policies
  • Verify users can only read their own data
  • Verify users cannot modify other user's data
  • Admin should have elevated permissions with audit trails
  • Test RLS with unauthorized user attempts

API Rate Limiting Not Implemented

MEDIUM

Issue: No rate limiting on authentication endpoints

Risk: Brute force attacks on login, password reset, signup

✓ Remediation:
  • Implement rate limiting on Supabase edge functions
  • Limit login attempts: 5 per 15 minutes
  • Limit password reset: 3 per hour
  • Limit signup: 10 per hour per IP
  • Implement progressive delays (exponential backoff)

Insufficient Logging & Audit Trails

MEDIUM

Issue: No audit logs for security-critical operations

Risk: Cannot detect intrusions or track unauthorized access

✓ Remediation:
  • Log all authentication events (login, logout, signup)
  • Log data modifications by user and timestamp
  • Log admin actions with user identification
  • Implement alerting for suspicious patterns
  • Retain audit logs for minimum 1 year
