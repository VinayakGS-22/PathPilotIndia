📋 Repository Overview

PathPilotIndia is a comprehensive career and education guidance platform built with React, TypeScript, and Vite. It's designed to help students in India navigate their educational and career paths by providing recommendations for colleges, scholarships, exams, careers, and mock tests based on their education stage and interests.

🔐 Authentication Flow

PathPilotIndia uses a comprehensive authentication system built on Supabase with a localStorage fallback mechanism for development and offline scenarios.

Sign-Up Flow

1
User navigates to SignupPage and enters email, password, name, role, and education stage
2
AuthContext.signUp() is called with all parameters
3
IF Supabase Configured: supabase.auth.signUp() creates auth user with email verification
4
Check if email is founder (gsvinayak8@gmail.com) → auto-assign admin role
5
Insert profile record into profiles table with role, education_stage
6
Email verification sent to user
7
User redirected to /verify-email page

Sign-In Flow

1
User navigates to LoginPage and enters credentials
2
IF Supabase Configured: supabase.auth.signInWithPassword() validates credentials
3
ELSE: Check against DEMO_USERS array or localStorage
4
onAuthStateChange listener detects 'SIGNED_IN' event
5
fetchProfile() loads user's profile from database
6
buildUserFromProfile() constructs complete User object
7
User state set globally via AuthContext
8
App redirects based on user state: /verify-email → /onboarding → /dashboard

Protected Route Logic

1
User attempts to access protected route
2
ProtectedRoute component checks AuthContext state
3
IF loading: Display loading spinner
4
IF !user: Redirect to /login
5
IF requireAdmin && user.role !== 'admin': Redirect (Unauthorized)
6
All checks pass → Render protected content

🔍 Unused Code Analysis

This section identifies redundant, unused, or optimizable code patterns that could be refactored or removed to improve code quality and bundle size.

1. Demo Users Array

src/contexts/AuthContext.tsx (lines 33-37)
const DEMO_USERS = [
  { email: 'student@demo.com', password: 'demo123', role: 'student' as UserRole },
  { email: 'parent@demo.com', password: 'demo123', role: 'parent' as UserRole },
  { email: 'admin@pathpilot.com', password: 'admin123', role: 'admin' as UserRole },
];

MEDIUM PRIORITY

Status: Used only in signIn() function when Supabase is not configured

Issues:

  • Hardcoded test credentials in production code
  • Only used during development/testing
  • Poses security risk if not removed for production builds

Recommendations:

  • Move to separate testing utility file
  • Exclude from production builds using environment checks
  • Use environment variables for test credentials

2. localStorage Fallback System

src/contexts/AuthContext.tsx (multiple locations)

MEDIUM PRIORITY

Components:

  • loadFromLocalStorage() function (lines 76-91)
  • localStorage.getItem calls (lines 77, 244)
  • localStorage.setItem calls (lines 211, 260, 296, 347)
  • localStorage.removeItem calls (lines 88, 140, 273)

Issues:

  • Creates dual code paths (Supabase vs localStorage)
  • Increases cognitive load and maintenance burden
  • Mixed authentication mechanisms difficult to debug

Recommendations:

  • Consolidate authentication logic
  • Document which path is production vs development
  • Use environment variables to toggle authentication mode
  • If Supabase is mandatory in production, remove all localStorage persistence

3. Defensive Helper Functions

src/contexts/AuthContext.tsx (lines 39-46)
function resolveRole(email: string, dbRole: string | null): UserRole {
  if (email === FOUNDER_EMAIL) return 'admin';
  return (dbRole as UserRole) || 'student';
}

function resolveApproval(_email: string, dbApproval: string | null): ApprovalStatus {
  return (dbApproval as ApprovalStatus) || 'approved';
}

LOW PRIORITY

Issues:

  • Add defensive defaults but may indicate database inconsistencies
  • Fallback to 'student' role hides data quality issues
  • Fallback to 'approved' status bypasses validation

Recommendations:

  • Add database constraints: NOT NULL on role and approval_status
  • Implement data migration to populate null values
  • Simplify to direct type casting if database guarantees are in place

4. Large Constants File ⚠️ HIGH PRIORITY

src/lib/constants.ts (105.5 KB)

HIGH PRIORITY

Content: India-wide educational data including:

  • All Indian states and their info
  • Career profiles and roadmaps
  • Exam information
  • Scholarship listings
  • College recommendations

Issues:

  • Extremely large for a static file (105 KB uncompressed)
  • Increases bundle size by ~30-40%
  • All data loaded on app initialization
  • Cannot be updated without code changes
  • Difficult to maintain and version
  • Duplicates data that should come from database

Recommendations:

  • Priority 1: Migrate to Supabase database tables
  • Priority 2: Implement lazy-loading API endpoints
  • Priority 3: Keep only UI constants in this file
  • Expected Impact: 30-50% bundle size reduction

Migration Strategy:

  1. Create Supabase tables: careers, exams, scholarships, colleges
  2. Import constants.ts data and insert into tables
  3. Implement API client in lib/api.ts
  4. Replace context data loading from localStorage to API
  5. Add pagination and filtering support
  6. Remove constants.ts once migration complete
